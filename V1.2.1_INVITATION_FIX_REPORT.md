# V1.2.1 Invitation Fix Report

**Release:** v1.2.1  
**Branch:** `feature/v1.2.1-communication-stabilization`

## Summary

Fixed production user invitation failures that returned generic HTTP 500 errors. Invitations now return actionable validation/conflict messages, audit logging, and resilient email delivery.

## Changes

| Area | Fix |
|------|-----|
| Duplicate email detection | Check active, invited, and soft-deleted accounts |
| Error handling | Map Postgres `23505` → 422/409 with clear messages |
| Mail dispatch | Best-effort delivery logging; no false failures |
| Invitation email | Expiry date, support contact, branded template |
| Audit | `user.invited` audit entry with actor + target |
| Frontend | Display backend error messages (not generic 500 text) |

## Expected behaviour after fix

1. Admin invites user → `201 Created`
2. User row created with `INVITED` status
3. Audit log written
4. In-app notification created
5. Branded invitation email sent via relay
6. Delivery logged in `message_deliveries`
7. Clear error if email unavailable (422, not 500)

## Error messages (examples)

| Scenario | HTTP | Message |
|----------|------|---------|
| Duplicate active email | 422 | A user with this email already exists. |
| Pending invitation | 409 | An invitation is already pending for this email address. |
| Soft-deleted email | 409 | This email belongs to a deleted account... |
| Mail not configured | 422 | Email delivery is not configured. |
| Relay failure | 422 | Email relay is unavailable... |

## Status

✅ Root cause fixed and documented in `INVITATION_ROOT_CAUSE_ANALYSIS.md`
