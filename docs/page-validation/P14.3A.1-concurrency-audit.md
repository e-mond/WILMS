## Strategy

**Option B — Optimistic versioning** (consistent with P14.3A readiness audit).

| Resource | Column | Update pattern |
|----------|--------|----------------|
| `loans` | `version` | `WHERE id = ? AND version = expected` |

Failed updates throw `CONFLICT:…` → HTTP 409 via `mapFinancialRouteError()`.

---

## Test Methodology

`runConcurrentRepaymentCheck(n)` in `db-checks.ts`:

1. Target seed active loan (`…0001`)
2. Find next non-PAID schedule week
3. Verify payment date weekday matches loan payment day
4. Fire **N parallel** `recordPayment()` calls (unique idempotency keys per attempt)
5. Assert: **≤1 success**, **≤1 new payment row**

Concurrency levels: **10, 25, 50**.

---

## Expected Outcomes

| Outcome | Meaning |
|---------|---------|
| 1 × success | Winning transaction posts payment |
| N−1 × CONFLICT/VALIDATION/DUPLICATE | Losers fail on version or no obligation |

All concurrent attempts target the **same oldest payable week** — only one schedule version increment can succeed.

## Retry Behavior

Clients receiving 409 should retry with fresh read. Idempotency keys on payment POST allow safe client retries after success.

Failed idempotency attempts delete IN_PROGRESS key (`run-with-idempotency.ts`) enabling retry after transient failure.
