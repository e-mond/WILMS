## Provider

**Neon PostgreSQL** — production database behind Railway `DATABASE_URL`.

Neon provides automatic continuous backup with **Point-in-Time Recovery (PITR)** on paid plans. Retention window depends on Neon plan tier (typically 7–30 days).

---

## Documented Restore Procedure

### Prerequisites

- Neon dashboard access (project owner or admin)
- Target restore timestamp (UTC)
- Approval for production restore (STOP GATE — production downtime)

### Steps

1. **Neon Console** → select WILMS production project/branch.
2. Navigate to **Branches** → production branch → **Restore**.
3. Choose **Point in time** and enter timestamp **before** the incident (e.g. before bad migration).
4. Neon creates a **new branch** at that point — does not overwrite production in place.
5. Copy new branch connection string; verify schema + row counts on restored branch.
6. **Swap connection:**
   - Update Railway `DATABASE_URL` to restored branch, **or**
   - Promote restored branch to primary (Neon dashboard) after validation.
7. Redeploy API; confirm `/health` → `database.connected: true`, `migrations.status: ok`.
8. Run `npm run smoke:production`.

### Rollback without restore

For additive migrations only (e.g. `0007_offline_sync`): run documented SQL down migration instead of PITR.
