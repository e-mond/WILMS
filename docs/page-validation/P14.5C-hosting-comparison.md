# P14.5C — Hosting Comparison

**Date:** 2026-06-29

---

## Frontend + backend options

### Option A — Vercel + Railway + Neon + Cloudinary (RECOMMENDED)

| Dimension | Assessment |
|-----------|------------|
| **Advantages** | Next.js native on Vercel; Express persistent runtime on Railway; Neon already integrated; minimal config (`vercel.json` + `railway.toml`); independent scale; BFF stays on Vercel |
| **Disadvantages** | Two platforms to manage; two billing accounts; cross-service env var wiring |
| **Cost (estimate)** | Vercel Pro ~$20/mo + Railway Hobby ~$5/mo + Neon free tier + Cloudinary free tier ≈ **$25–40/mo** at low traffic |
| **Scalability** | Vercel auto-scales frontend; Railway horizontal scaling available; Neon serverless scales connections |
| **Cold starts** | Vercel: minimal for SSR; Railway: **yes** on free/hobby if service sleeps — use always-on plan for API |
| **Security** | TLS at both edges; secrets per platform; BFF hides backend URL from browser |
| **CI/CD** | Vercel Git integration + Railway GitHub deploy; GitHub Actions template provided |
| **Maintainability** | **Best** — matches framework strengths |

### Option B — Railway (frontend + backend) + Neon + Cloudinary

| Dimension | Assessment |
|-----------|------------|
| **Advantages** | Single vendor; unified logs; `render.yaml`-style simplicity on one dashboard |
| **Disadvantages** | Next.js on Railway less optimized than Vercel; longer builds; no Vercel edge features |
| **Cost** | ~$10–20/mo (two Railway services) + Neon + Cloudinary |
| **Scalability** | Good for API; frontend SSR heavier than Vercel |
| **Cold starts** | Both services may sleep on hobby tier |
| **Security** | Same as A if BFF pattern preserved |
| **CI/CD** | Single Railway project |
| **Maintainability** | Good — acceptable if team prefers one vendor |

### Option C — Render (frontend + backend) + Neon + Cloudinary

| Dimension | Assessment |
|-----------|------------|
| **Advantages** | `render.yaml` blueprint in repo; predictable pricing; health checks built-in |
| **Disadvantages** | Slower cold starts on free tier; Next.js monorepo build more manual |
| **Cost** | ~$14/mo (2× Starter) + Neon + Cloudinary |
| **Scalability** | Adequate for V1 |
| **Cold starts** | **Notable** on free/starter — 30s+ possible |
| **Security** | Adequate with TLS + env secrets |
| **CI/CD** | Render auto-deploy from Git |
| **Maintainability** | Moderate |

### Option D — Monolithic (single container / single service)

| Dimension | Assessment |
|-----------|------------|
| **Advantages** | One deploy artifact; one URL; simpler local parity |
| **Disadvantages** | No Dockerfile exists; must run `next start` + `tsx` API together; couples release cycles; defeats Vercel BFF model |
| **Cost** | One VM/container ~$10–25/mo |
| **Scalability** | Poor — must scale entire stack together |
| **Cold starts** | N/A (always-on VM) or high (serverless monolith refactor) |
| **Security** | Single attack surface |
| **CI/CD** | Custom Docker pipeline required |
| **Maintainability** | **Poor** for current codebase — not recommended |

## Backend-only platform comparison (for API host)

| Platform | Fit for WILMS Express | Notes |
|----------|----------------------|-------|
| **Railway** | **Best** | npm workspace monorepo; `PORT` injection; health checks; simple Git deploy |
| **Render** | Good | `render.yaml` provided; cold starts on starter |
| **Fly.io** | Good | Needs Dockerfile (not in repo); more ops overhead |
| **Azure App Service** | Adequate | Enterprise overhead; overkill for V1 |
| **DigitalOcean App Platform** | Adequate | Similar to Render; monorepo config manual |

**Recommendation:** **Railway** for `@wilms/api`.

---

## Final recommendation

# Option A

**Vercel** (frontend + BFF) + **Railway** (Express API) + **Neon** + **Cloudinary**

Simplest path to production that respects the existing architecture without rewriting the backend for serverless.
