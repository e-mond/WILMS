## Potential breaking changes (for P14.1C+)

| # | Risk | Evidence |
|---|------|----------|
| 1 | Align `PaymentTransaction.status` values | Frontend enum vs backend `RECORDED` |
| 2 | Expand `PaymentEntryContext` backend or simplify frontend | `payment-entry.ts` vs stub route |
| 3 | Unify `BorrowerReviewDetail` shape | Nested vs flat profile |
| 4 | Persist all registration upload IDs | Payload fields L90–95 vs `BorrowerProfile` |
| 5 | Add `gps.capturedAt` to backend GPS | `payment.ts` L5 |
| 6 | Implement payment PATCH mutation | Current edit is no-op on data |
| 7 | `BorrowerStatus` enum parity | 6 vs 4 values |
| 8 | Typed audit actions on backend | Free-form string today |
| 9 | Multipart upload contract | Breaking if removing `dataUrl` requirement |
| 10 | `full-profile` must return real loans | Currently stub breaks profile UI in API mode |
| 11 | Reviewed applications history | Empty backend list breaks approver reviewed page |
| 12 | Notification side-effects | UI may expect inbox entries after mutations |
