# P14.3B ‚Äî Reversal Safety Design

**Phase:** P14.3B.3B  
**Date:** 2026-06-21

---

## Design Goals

Every reversal operation must be protected by:

- Concurrency control (optimistic locking)
- Idempotency (safe retry)
- Audit (actor + reason + before/after)
- RBAC (existing permissions only)

Reuse P14.3A patterns ‚Äî **no second concurrency model**.

---

## Concurrency

### Locking strategy

| Resource | Mechanism | Evidence (existing) |
|----------|-----------|---------------------|
| Loan balance / lifecycle | `loans.version` optimistic lock | `loan.repository.updateLoanLifecycle` |
| Schedule week | `loan_schedules.version` | `scheduleRepo.markWeekPaid` |
| Payment row | `payments.version` (exists, unused on PATCH) | `payments` schema L26 |
| Adjustment row | `financial_adjustments.version` | `adjustment.repository.approveAdjustmentRow` |
| Reversal row | **New** `financial_reversals.version` | Mirror adjustment pattern |

### Transaction scope

All reversal effects in **one** `runInTransaction` ‚Äî same as `postPayment` and `approveAdjustment`.

Order of locks (deadlock avoidance):

```text
1. financial_reversals (insert)
2. payments / adjustments (source row, version check)
3. loans (version check)
4. loan_schedules (version check)
5. ledger_entries (append)
6. pool_allocations (append)
7. reversal_history (append)
```

### Version strategy

On conflict (`CONFLICT:*` message): return HTTP 409 via `mapFinancialRouteError` ‚Äî existing pattern.

### Retry behavior

| Scenario | Behavior |
|----------|----------|
| Optimistic lock failure | Client refreshes and retries with new version |
| Idempotency replay | Return cached response body ‚Äî no double reversal |
| Partial txn failure | Full rollback ‚Äî no orphan ledger row |

### Race scenarios

| Race | Mitigation |
|------|------------|
| Concurrent payment + reversal on same loan | Loan version ‚Äî second writer fails |
| Concurrent two reversals same payment | Unique constraint `(source_type, source_id)` on `financial_reversals` WHERE status != REJECTED |
| Concurrent adjustment approve + payment reversal | Check source payment not under pending adjustment |

## Idempotency

### Storage

Reuse `idempotency_keys` table ‚Äî proven in P14.3A.

### New scopes (design)

| Scope | Use |
|-------|-----|
| `REVERSAL_CREATE` | POST /reversals (if split request/execute in v2) |
| `REVERSAL_EXECUTE` | Combined create+execute v1 |

Add to `idempotency_scope` enum in 3C migration (design only).

### Key format

```text
Idempotency-Key: client-generated UUID
Unique identity: (scope, actorUserId, idempotencyKey)
Business identity: (sourceType, sourceId) enforced by DB unique partial index
```

### Replay behavior

Same as `runWithIdempotency`:

- `COMPLETED` ‚Üí return stored `responseBody`
- `IN_PROGRESS` ‚Üí 409 `IDEMPOTENCY_IN_PROGRESS`

### Duplicate detection

1. Idempotency header (transport retry)
2. DB unique index on `(source_type, source_id)` for terminal reversal states
3. Payment status check `CONFIRMED` only

### TTL

24 hours ‚Äî `RETENTION_MS` in `idempotency.repository.ts` (unchanged).

## Audit

### New audit actions (design ‚Äî enum extension in 3C)

| String key | Enum value | When |
|------------|------------|------|
| `reversal.requested` | `REVERSAL_REQUESTED` | Reversal row created |
| `reversal.approved` | `REVERSAL_APPROVED` | Two-step model only |
| `reversal.executed` | `REVERSAL_EXECUTED` | After successful txn |
| `reversal.rejected` | `REVERSAL_REJECTED` | Admin rejects request |

Extend `audit.repository.ts` `ACTION_MAP` and `audit_target_entity` with `REVERSAL`.

### Actor tracking

| Field | Source |
|-------|--------|
| `actorId` | Session user UUID |
| `actorDisplayName` | Request body (matches adjustment pattern) |
| `reason` | Required min 10 chars (matches adjustment) |

### Before / after values

Stored in **`reversal_history`** (transactional, authoritative for financial proof):

| Field | Description |
|-------|-------------|
| `before_balance_pesewas` | Loan balance before |
| `after_balance_pesewas` | Loan balance after |
| `delta_pesewas` | Signed change |
| `before_source_status` | e.g. payment CONFIRMED |
| `after_source_status` | payment REVERSED |

Audit table carries reason + actor; history table carries financial snapshots (mirror `adjustment_history`).

### Async audit caveat

Current `appendAuditEntry` is post-commit async ‚Äî **accepted for parity with payments/adjustments**.

**Risk R-SAFE-01:** Reversal audit not in same DB txn. Mitigation: `reversal_history` is transactional source of truth; audit is supplementary.

## RBAC

Use `@wilms/shared-rbac` permissions only ‚Äî no new roles.

### Proposed mapping

| Action | Permission | Roles (default) |
|--------|------------|-----------------|
| Request reversal | `ACCESS_ADMIN_PORTAL` | SUPER_ADMIN |
| Approve reversal (v2) | `ACCESS_ADMIN_PORTAL` | SUPER_ADMIN |
| Execute reversal (v1 combined) | `ACCESS_ADMIN_PORTAL` | SUPER_ADMIN |
| View reversals | `VIEW_FINANCIAL_REPORTS` | SUPER_ADMIN, AUDITOR |
| View audit | `VIEW_AUDIT_LOG` | AUDITOR |

**Not** `RECORD_COLLECTIONS` for execute ‚Äî reversals are admin financial controls (aligns with adjustment approve).

**Same-day collector edit:** Mock uses collector edit + adjustment ‚Äî Phase 3C should **not** expose reversal execute to collectors without explicit product change.

### Route guards

```text
requireAuth + requirePermission(...)  -- same middleware as adjustments/routes.ts
```

## Safety Exit Criteria

| Control | Designed |
|---------|----------|
| Concurrency | Yes ‚Äî optimistic version on all mutated rows |
| Idempotency | Yes ‚Äî REVERSAL_EXECUTE scope + source unique index |
| Audit | Yes ‚Äî enum + history snapshots |
| RBAC | Yes ‚Äî ACCESS_ADMIN_PORTAL execute, VIEW_FINANCIAL_REPORTS read |

**Phase 3 exit:** PASS
