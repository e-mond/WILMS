## Business Purpose

Implements the backend business layer for **collector cash reconciliation MVP**: compute expected due and system-recorded totals, classify variance, persist immutable submissions with append-only history, idempotency, and audit — per `P14.3B-reconciliation-architecture.md`.

**No HTTP routes, frontend, or certification harness in this phase.**

## Service Workflow (`submitReconciliation`)

```text
1. requireDatabase()
2. runWithIdempotency(RECONCILIATION_SUBMIT)
3. Check existing submission → 422 VALIDATION
4. Load portfolio loans + payments for date
5. buildReconciliationSnapshot()
6. Validate comment if variance_flagged (min 10 chars)
7. runInTransaction:
   a. insertReconciliation
   b. appendReconciliationHistory (SUBMITTED)
8. appendAuditEntry (post-commit, async best-effort)
9. Return ReconciliationSummary
```

Also exposes: `getReconciliationSummary`, `getReconciliationById`, `listReconciliations`.
