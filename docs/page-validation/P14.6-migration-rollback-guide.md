# P14.6 — Migration Rollback Guide

**Release:** v0.2.2  
**Migration:** `0007_offline_sync`  
**Date:** 2026-06-30

---

## Up (apply)

```bash
npm run db:migrate -w @wilms/api
```

Creates:

- `offline_sync_operations` + unique idempotency index + status index
- `offline_sync_conflicts` + status index

Journal: `apps/backend/src/db/migrations/meta/_journal.json` entry idx 7.

**Production:** Run via `deploy-production.yml` migration step or Railway pre-deploy hook.

---

## Down (rollback)

SQL (from migration file footer):

```sql
DROP TABLE IF EXISTS offline_sync_conflicts;
DROP TABLE IF EXISTS offline_sync_operations;
```

### Manual rollback steps

1. **Stop API** or enable maintenance mode (prevent new sync ingests).
2. **Export pending conflicts** if review in flight:
   ```sql
   SELECT * FROM offline_sync_conflicts WHERE status = 'PENDING_REVIEW';
   SELECT * FROM offline_sync_operations WHERE status = 'QUEUED_FOR_REVIEW';
   ```
3. Run down SQL on production Neon (via psql or Neon SQL editor).
4. Remove journal entry `0007_offline_sync` from `drizzle.__drizzle_migrations` **only if** re-running migrate later — otherwise Drizzle will attempt re-apply.
5. Redeploy API build **without** sync routes (revert to v0.2.1 tag).
6. Verify `/health` → `migrations.applied: 7`.

---

## Data Impact

| Action | Data loss |
|--------|-----------|
| Down migration | **All queued offline operations and conflicts deleted** |
| Up re-apply | Empty tables — clients must re-sync |

No impact on `payments`, `loans`, or other financial tables if conflicts were not approved.

---

## When to Rollback

- Sync ingest causing production errors
- Schema conflict with unexpected Drizzle state
- Decision to defer offline sync to v0.2.3

**Prefer PITR** (see `P14.6-backup-verification.md`) if financial data corruption suspected — not for clean sync-only rollback.

---

## Verdict

Rollback path documented. Down migration is destructive to sync queue only — safe if no approved offline payments pending reconciliation.
