## Idempotency

| Layer | Mechanism |
|-------|-----------|
| **DB** | Unique index `(actor_user_id, idempotency_key)` on `offline_sync_operations` |
| **Ingest** | Pre-insert lookup; existing row → `DUPLICATE` with same `operationId` |
| **Payment apply** | `approveConflict` passes `operation.idempotencyKey` to `paymentService.recordPayment` |
| **Ordering** | Batch sorted by `clientCreatedAt` before processing |

Duplicate replays are safe: no second conflict row, no double payment.

## Financial Conflict Queue

| Constant | Value |
|----------|-------|
| `FINANCIAL_OPERATION_TYPES` | `Set(['RECORD_PAYMENT'])` |

**Ingest path (financial):**

1. `runInTransaction` inserts operation with `status: QUEUED_FOR_REVIEW`.
2. Inserts conflict with `status: PENDING_REVIEW`, reason: *"Financial mutation from offline queue — manual review required."*
3. Returns `QUEUED_FOR_REVIEW` + `conflictId`.

**Approve path:**

1. Loads conflict + operation inside transaction.
2. For `RECORD_PAYMENT`, delegates to `paymentService.recordPayment` (existing validation, GPS, duplicate checks).
3. Updates conflict → `RESOLVED`, operation → `APPLIED` with `result.payment`.

**Reject path:**

1. Conflict → `REJECTED`, operation → `REJECTED`.
2. No payment side effects.
