# P14.4 — Executive Summary (Stakeholder)

**Date:** 2026-06-20  
**Audience:** Non-technical stakeholders  
**Verdict:** **NOT READY** for production (see P14.4I)

---

## What is in Version 1

WILMS Version 1 is defined as a **financial-core release**. The system supports:

- **Collectors** recording payments and submitting daily cash reconciliation (certified workflows).
- **Registration officers** registering new borrowers with KYC checks and photo uploads.
- **Approvers** reviewing and approving borrowers and loans, including disbursement.
- **Super admins** managing loans, borrowers, financial adjustments, loan pools, and viewing reports (with some report tables showing empty data until populated).
- **Auditors** viewing audit logs and reports.

Six user roles exist: Super Admin, Collector, Registration Officer, Approver, and Auditor. Authentication uses secure login sessions (no OTP or password reset in V1).

**25 of 46 application pages** are approved to ship in V1. **20 pages** (admin dashboard, groups, collector management, settings, risk flags, and similar) are deferred to Version 2 because they only work in demo/mock mode today — they have no live backend.

## What is not in Version 1

Deferred to Version 2 (safe to postpone for financial operations):

- Full groups and collector administration
- User/settings management screens (live API)
- Super-admin dashboard summary
- Risk flags engine
- SMS and email notification workflows
- OTP and password self-service reset
- Loan write-offs and advanced pool features
- Five detailed reports that currently return empty data

See `P14.4H-version2-backlog.md` for the full prioritized list (32 items).

## Known limitations

| Limitation | Mitigation |
|------------|------------|
| Admin pages work only in demo mode | Use financial-core pages for live operations; V2 adds APIs |
| Cloudinary cloud uploads not configured | Local file uploads work; staging needs credentials |
| Some reports show empty tables | Daily collection report works; others planned for V2 |
| OTP / password reset not built | Manual admin processes |
| Health check does not test database | Ops must monitor database separately until fixed |

Accepted from prior certification: group-formation permission gap (low traffic); security headers via reverse proxy.

## Release recommendation

**NOT READY** — production deployment must not proceed.

The financial core is **built and certified with conditions**, but **operational blockers** (cloud uploads, HTTPS, merge to main, human approval, and integration test gaps) remain unresolved. After blockers are addressed, the project may move to **Ready with Conditions** for a staging deployment of the financial-core scope.
