## Risk Register

| ID | Risk | Severity | Evidence | Mitigation (3B design) |
|----|------|----------|----------|------------------------|
| R1 | Double financial effect (adjustment + reversal) | **Critical** | UI uses `PAYMENT_CORRECTION` adjustment; Phase 3 reversals are compensating entries | Mutual exclusion rules; source-txn idempotency |
| R2 | Payment PATCH bypasses Neon | **High** | `payments/routes.ts` PATCH always uses in-memory `listPayments` | Unify on `paymentService` or dedicated reversal service |
| R3 | Schedule week remains PAID after reversal | **High** | `markWeekPaid` on record; no unpay path | Reversal txn must revert schedule week status |
| R4 | Pool aggregates drift | **High** | Append-only `pool_allocations` | Compensating allocation on reversal approve |
| R5 | No ledger link to reversed entry | **Medium** | `ledger_entries` has no `reverses_entry_id` | Schema extension in 3B |
| R6 | Audit not transactional | **Medium** | `appendAuditEntry` async | Document acceptance or sync audit in reversal txn |
| R7 | Neon latency on multi-step reversal txn | **Medium** | Phase 0 baseline FAIL on schedule/list | Size reversal txn; index review in 3B |
| R8 | Frontend expects REVERSAL transaction type | **Medium** | `TRANSACTION_TYPE` has no REVERSAL | Define contract in 3B before 3C implementation |
| R9 | `verify:financial` seed mutation | **Low** | Adjustment harness modifies seed loan balance | Isolate reversal test fixtures |
| R10 | Doc stale (adjustments "mock only") | **Low** | Older P14.3A.2 docs | Governance update outside 3A scope |

## Recommended 3B Architecture Topics

1. Reversal entity schema vs event-sourced ledger-only model.
2. Payment reversal vs disbursement reversal vs pool allocation reversal scopes.
3. Approval gate: immediate collector same-day vs admin-approved (align with mock UX).
4. Frontend API contract (`/reversals` vs extend `/payments/:id/reverse`).
5. Interaction with `PAYMENT_CORRECTION` adjustments — deprecate or gate.
