## Recovery procedures (documented)

### Demo / cert environment reset

```bash
npm run cert:reconciliation:seed-reset -w @wilms/api
npm run cert:reversal:seed-reset -w @wilms/api
npm run db:seed -w @wilms/api
# Restart API — required for bcrypt password hashes
```

### Database (Neon)

- Point-in-time recovery available via Neon console (5A guide).
- **Restore drill:** not executed in 5B session.

### Uploads (local provider)

- Files under `WILMS_UPLOAD_DIR` (`.wilms-uploads`).
- Loss of disk = re-upload required; metadata in PostgreSQL.

### Uploads (Cloudinary — production target)

- Outage handling documented in cloudinary-setup; **not exercised**.

## Failures & blockers

| Risk | Severity | Gap |
|------|----------|-----|
| Silent DB failure | High | Health does not fail when DB unreachable |
| No tested Cloudinary fallback | Medium | Production uploads depend on external SLA |
| Shared Neon cert pollution | Medium | DR for "clean cert state" requires seed-reset discipline |

---

## Summary

**Process restart** and **session invalidation** behaviors are acceptable. **Infrastructure failure detection** (DB, Cloudinary) is **incomplete**. Demo recovery documentation is **adequate** for stakeholder rehearsals; production DR requires health hardening and backup restore drill.
