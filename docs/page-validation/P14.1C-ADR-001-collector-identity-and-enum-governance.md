# Decision 1: Collector Identity Model

## Approved Architecture

Collector is a specialized User.

Collectors SHALL NOT maintain a separate authentication identity.

Authentication, authorization, auditing, notifications, sessions, and permissions must originate from the Users domain.

---

## Why This Decision Was Chosen

During P14.1A and P14.1B audits, the system already demonstrated:

- Centralized RBAC
- Centralized permission checks
- Centralized session management
- Centralized authentication
- Role-driven navigation
- Role-driven workflows

Creating a separate Collector identity would duplicate:

- Login credentials
- Session management
- Permission management
- Audit ownership
- Notification ownership
- Upload ownership

This would increase complexity without providing any architectural benefit.

---

## Approved Model

```text
users
 ├── id (PK)
 ├── email
 ├── passwordHash
 ├── roleId
 ├── status
 ├── createdAt
 └── updatedAt

collectors
 ├── id (PK)
 ├── userId (FK → users.id)
 ├── collectorCode
 ├── assignedRegion
 ├── assignedDistrict
 ├── employmentStatus
 ├── createdAt
 └── updatedAt
```

Relationship:

```text
User 1 ─── 0..1 Collector
```

A collector profile cannot exist without a corresponding user account.

---

## Benefits

### Authentication

Single login source.

### RBAC

Single permission source.

### Audit Logging

Every action references `users.id`.

### Notifications

Single recipient model.

### Upload Ownership

Uploads can be owned by users.

### Future SSO/MFA

Implemented once at User level.

### Reporting

Collectors remain reportable while retaining user identity.

---

## Non-Approved Alternative

Separate Collector Authentication Entity.

Reasons rejected:

- Duplicated identity storage
- Duplicate RBAC implementation
- Duplicate session management
- Increased audit complexity
- More migration work
- No demonstrated business benefit

# Decision 2: Enum Governance

## Approved Architecture

All application enums shall be maintained from a single canonical backend source.

Frontend enums must be generated or synchronized from backend contracts.

---

## Backend Source of Truth

```text
backend/src/contracts/enums/
```

Examples:

- BorrowerStatus
- LoanStatus
- PaymentStatus
- GroupStatus
- ApprovalStatus
- NotificationStatus
- UploadPurpose
- RiskStatus
- ExpenseStatus
- ReconciliationStatus
- UserStatus

---

## Rule

No frontend-only enum may diverge from backend enum definitions.

Any enum change requires:

1. Backend contract update
2. API validation update
3. Frontend contract update
4. E2E validation

# Architectural Outcome

WILMS adopts:

- Single Identity Model
- Role-Based Access Control
- Collector-as-User Architecture
- Centralized Enum Governance
- Shared Audit Ownership
- Shared Notification Ownership
- Shared Upload Ownership

This decision is effective immediately and supersedes any future proposal that introduces standalone collector authentication records.
