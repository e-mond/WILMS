## Objective 4 — CSRF Assessment (STOP GATE 3)

### 1. Attack surface

**Browser-session mutating API routes (Express):**

| Module | Methods | Examples |
|--------|---------|----------|
| auth | POST | `/auth/login` |
| loans | POST, PATCH | create, approve, disburse |
| payments | POST, PATCH | record, reverse |
| adjustments | POST | create, approve |
| borrowers | POST, PATCH, DELETE | register, update |
| uploads | POST | upload metadata |
| reconciliation | POST | submit |
| group-formation | POST | create |
| audit | POST | append |

**BFF (Next.js):** `/api/auth/login`, `/api/auth/logout`, `/api/wilms/*` proxy all methods.

### 2. Current mitigation (`SameSite=Lax`)

- **Protects:** Top-level navigations; cookie not sent on cross-site POST from third-party origins in most cases.
- **Does not protect:** Cross-site requests if attacker uses top-level form POST to same site; some legacy browsers; attacks via same-site subdomains; state-changing GET (none identified as primary pattern).

Auth uses **Bearer token in HttpOnly cookie** converted to `Authorization` header by BFF (`route.ts:26-28`).

### 3. Risk level: **Medium**

Financial mutations (payments, reversals, loan approval) are high impact. SameSite=Lax reduces classic CSRF but is not a complete control for a session-cookie BFF architecture.

### 4. Implementation options

| Option | Complexity | Notes |
|--------|------------|-------|
| **Synchroniser token** | Medium | Issue CSRF token on login; validate on BFF mutating routes before proxy |
| **Double-submit cookie** | Low–Medium | Non-HttpOnly CSRF cookie + header match on BFF |
| **SameSite=Strict** | Low | Breaks OAuth/deep links; may hurt UX |

### 5. Recommendation

**Defer implementation** for immediate go-live **if** demo credentials are replaced and deployment protection remains off only for production domain. **Implement synchroniser token on BFF** before stakeholder financial operations in production.

**⛔ No CSRF code written — awaiting decision.**
