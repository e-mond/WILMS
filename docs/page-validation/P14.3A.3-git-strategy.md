## Branch Rules

| Rule | Detail |
|------|--------|
| `main` | Protected; merge via PR only; must pass CI gates |
| `develop` | Default integration target for features |
| `feature/*` | Branch from `develop`; rebase or merge `develop` before PR |
| `hotfix/*` | Branch from `main`; merge back to `main` and `develop` |
| `release/*` | Branch from `develop`; only fixes/docs; merge to `main` + `develop` |

---

## Pull Request Rules

1. **Scope:** One roadmap phase or logical unit per PR
2. **Validation:** `type-check`, `lint`, `build`, `test` must pass
3. **E2E:** Required for UI/integration phases; `CI=1 npm run test:e2e`
4. **Docs:** Phase deliverables under `docs/page-validation/`
5. **No secrets:** `.env` never committed; examples only
6. **Review:** At least one approval before merge to `develop`/`main`

---

## Merge Strategy

| Target | Strategy |
|--------|----------|
| `feature → develop` | Squash merge (clean history) |
| `develop → main` | Merge commit (preserve release boundary) |
| `hotfix → main` | Merge commit + tag |

---

## Release Strategy

1. Cut `release/vX.Y.Z` from `develop`
2. Run full certification (migrations, seed, `verify:financial`, E2E)
3. Fix-only commits on release branch
4. Merge to `main`, tag `vX.Y.Z`
5. Merge `main` back to `develop`
