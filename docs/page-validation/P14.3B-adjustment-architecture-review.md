## Frontend Contract (source of truth)

**Service:** `apps/frontend/src/services/adjustmentService.ts`

| Method | HTTP | Body |
|--------|------|------|
| `listPendingAdjustments` | `GET /adjustments/pending` | — |
| `createAdjustment` | `POST /adjustments` | `CreateAdjustmentInput` + `actorId`, `actorDisplayName` |
| `approveAdjustment` | `POST /adjustments/:id/approve` | `{ actorId, actorDisplayName }` |
| `rejectAdjustment` | `POST /adjustments/:id/reject` | `RejectAdjustmentInput` + actors |

**Types:** `apps/frontend/src/types/adjustment.ts`  
**Types enum:** `PAYMENT_CORRECTION`, `DISBURSEMENT_CORRECTION`, `WRITE_OFF`, `BALANCE_ADJUSTMENT`  
**Status enum:** `PENDING`, `APPROVED`, `REJECTED`

**Mock approve side-effects** (`adjustmentService.mock.ts`):
1. Append `FinancialTransaction` type `ADJUSTMENT`
2. Audit: `ADJUSTMENT_REQUESTED`, `ADJUSTMENT_APPROVED`, `ADJUSTMENT_REJECTED`
3. `WRITE_OFF`: blacklist borrower + set loan `WRITTEN_OFF`

## Classification Matrix

| Concern | Classification | Decision |
|---------|----------------|----------|
| **Adjustment ownership** | **Safe** | Collector creates via `RECORD_COLLECTIONS` (payment correction UI); Super Admin approves via `ACCESS_ADMIN_PORTAL` / full permissions |
| **Adjustment lifecycle** | **Safe** | PENDING → APPROVED/REJECTED; only pending reviewable (mock enforces) |
| **Reversal interaction** | **Needs Alignment** | Phase 3 reversals are compensating ledger entries; adjustments are approval-gated corrections — must not double-apply |
| **Audit implications** | **Safe** | Enum values exist: `ADJUSTMENT_REQUESTED/APPROVED/REJECTED`, target `ADJUSTMENT` — wire `appendAuditEntry` + ACTION_MAP |
| **Reconciliation implications** | **Needs Alignment** | Reconciliation compares collector cash vs payments; adjustments must not silently change payment totals without reconciliation flag |
| **Write-off implications** | **Breaking Risk** | `WRITE_OFF` type overlaps Phase 5 write-offs — Phase 2 implements mock parity; Phase 5 may split dedicated write-off table |
| **Ledger impact** | **Needs Alignment** | Current ledger types: no `ADJUSTMENT` entry type — must extend `ledger_entry_type` enum + append-only entries on approve |
| **Pool impact** | **Needs Alignment** | Approved adjustments should post `pool_allocations` type `ADJUSTMENT` when loan has `loan_pool_id` |
| **Reporting impact** | **Safe** | `/reports/financial-ledger` expects `ADJUSTMENT` transaction rows |

## Risk Register

| Risk | Type | Mitigation |
|------|------|------------|
| Double ledger post on approve retry | Concurrency | Idempotency scope `ADJUSTMENT_APPROVE` + transaction |
| WRITE_OFF without pool adjustment | Financial | Post pool `ADJUSTMENT` allocation reducing outstanding |
| Partial payment correction vs reversal | Financial | PAYMENT_CORRECTION creates ledger compensating entry, not payment row mutation |
| Neon latency on pending list | Performance | Index on `(status, requested_at)` |
| Enum migration failure | Migration | Separate migration file; test on Neon before merge |

## Phase 2 Scope Boundary

**In scope:**
- Schema + migration
- CRUD workflow (create, list pending, approve, reject)
- Ledger append on approve (new entry type)
- Audit entries
- `verify:adjustments` harness

**Out of scope (later):**
- Dedicated write-off service (Phase 5)
- Reversal engine (Phase 3)
- Pool auto-allocation on every disburse (Phase 2+ pool hook optional)
