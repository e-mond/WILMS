# P13.2 Responsive Stability Audit

**Date:** 2026-06-15  
**Breakpoints referenced:** 320px, 360px, 390px, 414px, 768px (via Playwright projects and `@ mobile-375` / `@ tablet-768` suites)

## Method

- Mapped v3 E2E responsive failures to shell architecture in `DashboardShell`, `OfficeShell`, and `OperationalMobileNavigation`.
- Verified mobile navigation patterns per role in code (not manual device lab in P13.2).
- Fixed E2E shell expectations to match implemented responsive behavior.

## v3 Failures — Root Cause

| Failure | Root cause | P13.2 fix |
| ------- | ---------- | --------- |
| `registration-officer` / `approver` @ mobile-375 | Tests called `getByRole('button', { name: 'Open navigation menu' })` but operational roles **do not render** `MobileSidebarTrigger` (`enableMobileNavDrawer={false}`). | `expectOfficeShellAtBreakpoint` checks `{navAriaLabel} bottom navigation` and `role="tab"` primary item. |
| Super-admin @ mobile-375 (flaky) | Timing after sign-in + drawer FAB overlap | Overlay dedupe; unchanged drawer UX |
| Shell mobile aside tests | Duplicate dialogs + `table` selector matching layout tables outside `main` | Singleton overlays; `main table` scoped assertion |

## Verified Layout Constraints (no redesign)

- **Clipped actions:** Bottom nav uses `min-h-[56px]` tabs (`ShellNavLink` `tabMode`) — meets touch target sizing in code.
- **Hidden controls:** Desktop-only `AppNavbar` (`hidden md:block`) intentionally defers search to center trigger / mobile overflow — not a defect.
- **Overlapping drawers:** Single instance of search/notification modals after `ShellOverlayPanels` dedupe.
- **Broken navigation:** Approver E2E updated to `Approver Navigation` aria label (matches `ApproverShell.tsx`).
- **Layout shifts:** No loading-state changes in P13.2 (E2E waits on stable content markers).
