# P14.3B Phase 3C.2 — Performance Root Cause Analysis

**Date:** 2026-06-21  
**Scope:** Payment reversal latency (discovery only — **no optimizations applied**)  
**Measured baseline:** Batch 100 — avg **7382.5 ms**, p95 **10568.2 ms**, p99 **17272.1 ms** (`P14.3B-phase-3c2-performance-output.txt`)  
**Method:** Code-path inventory + certification harness methodology; counts are **static analysis from repository**, not query-log tracing.

---

## Executive Summary

Measured reversal latency (~7.4 s average on Neon) is dominated by **many sequential database round trips** across a **single remote Neon serverless connection**, not by CPU-bound application logic. The reversal path performs **5 pre-transaction reads**, then **one multi-statement transaction** with **~9 writes and ~3+ reads**, followed by **2 fire-and-forget audit inserts**. The performance harness adds substantial **setup cost outside the timer** (payment recording with repeated schedule/context queries). Neon **pooler latency**, **WebSocket transport**, and **long-session connection drops** are operational contributors observed during batch 500/1000 attempts.

## Finding RCA-01 — Pre-transaction read fan-out

| Field | Value |
|-------|-------|
| **File Path** | `apps/backend/src/modules/payments/payment-reversal.service.ts` L62–101 |
| **Evidence** | Before `runInTransaction`, `executePaymentReversal` calls: `findPaymentById`, `findExecutedReversalBySource`, `hasApprovedPaymentCorrectionConflict`, `findLoanById`, `findRepaymentLedgerByPaymentId` |
| **Observation** | Five separate SELECT round trips to Neon occur **outside** the atomic transaction |
| **Impact** | Each round trip adds remote latency (~100–500+ ms per hop on serverless); ~5× before txn begins |
| **Recommendation** | Document only (3C.2); future optimization could consolidate validation reads — **not in scope now** |

## Finding RCA-05 — High write count per reversal (transaction scope)

| Field | Value |
|-------|-------|
| **File Path** | `payment-reversal.service.ts` L112–241 |
| **Evidence** | Inside `runInTransaction`: insert reversal → 3× history inserts → payment update → schedule revert → loan update → ledger insert → reversal execute update → 2× history inserts |
| **Observation** | **~9 write statements** in one transaction (3 reversal_history, 1 financial_reversals insert + 1 update, 1 payments, 1 loan_schedules, 1 loans, 1 ledger_entries) |
| **Impact** | Long transaction hold time; each statement is a server round trip within Neon serverless driver |
| **Recommendation** | Correct for financial atomicity; latency is expected cost of append-only design |

### Transaction scope diagram

```text
executePaymentReversal
├── PRE-TXN READS (5 queries, separate connections)
│   payments, financial_reversals, financial_adjustments, loans, ledger_entries
├── runInTransaction (single BEGIN…COMMIT)
│   READ  financial_reversals (duplicate re-check)
│   WRITE financial_reversals INSERT
│   WRITE reversal_history INSERT (CREATED)
│   WRITE payments UPDATE → REVERSED
│   READ  loan_schedules SELECT * (all weeks)
│   READ  loan_schedules SELECT one week  ← revertWeekPaid
│   WRITE loan_schedules UPDATE
│   WRITE loans UPDATE (balance + lifecycle)
│   WRITE ledger_entries INSERT (REVERSAL)
│   WRITE financial_reversals UPDATE (EXECUTED)
│   WRITE reversal_history INSERT (EXECUTED)
│   WRITE reversal_history INSERT (LEDGER_POSTED)
└── POST-TXN (async, 2× fire-and-forget)
    audit_entries INSERT ×2
```

## Finding RCA-06 — Database reads/writes per reversal (inventory)

| Phase | Reads (min) | Writes (min) |
|-------|-------------|--------------|
| Pre-transaction validation | **5** | 0 |
| Idempotency (`REVERSAL_EXECUTE` key) | **1–2** (begin insert + complete update) | **1–2** |
| Inside transaction | **2–3** (duplicate check + full schedule + week re-select) | **~9** |
| Post-commit audit (async) | 0 | **2** |
| **Total (typical perf cert iteration)** | **~8–10** | **~12–13** |

*Idempotency:* perf harness passes unique key each iteration (`cert-reversal-performance.ts` L110) → always inserts new idempotency row.

## Finding RCA-07 — N+1 in payment setup (harness path, outside timer)

| Field | Value |
|-------|-------|
| **File Path** | `apps/backend/src/verification/reversal-checks.ts` L202–211; `payments/service.ts` L149–150; `loan-schedule.repository.ts` L119–139 |
| **Evidence** | `recordReversiblePayment` loops unpaid `referenceDates` and calls `getPaymentEntryContext` per date; each call runs `applyMissedWeekMarking` which loops all weeks with **one UPDATE per overdue PENDING week** |
| **Observation** | Payment setup for perf iteration can trigger **O(weeks × dates)** queries before timed reversal |
| **Impact** | Batch wall-clock time >> timed `reversePayment()` latency; not included in avg ms but affects total cert runtime and Neon session length |
| **Recommendation** | Document; cert methodology separates setup from timed reversal intentionally |

## Finding RCA-08 — Post-commit async audit (non-transactional)

| Field | Value |
|-------|-------|
| **File Path** | `apps/backend/src/infrastructure/audit/audit-log.ts` L41–44; `payment-reversal.service.ts` L243–259 |
| **Evidence** | `void auditRepository.appendAuditEntry(input).catch(...)` ×2 after commit |

| **Impact** | Extra connection usage; long batches may fail audit persist without failing reversal |
| **Recommendation** | Known from P14.3A safety audit (R-SAFE-01); transactional proof is `reversal_history` |

## Finding RCA-09 — Neon-specific latency contributors

| Field | Value |
|-------|-------|
| **File Path** | `apps/backend/src/db/client.ts` L1–41; `P14.3B-phase-3c2-env-probe-output.txt`; `P14.3B-phase-3c2-performance-500-1000-output.txt` |
| **Evidence** | `@neondatabase/serverless` Pool + WebSocket; host `ep-noisy-river-atkdh45c-pooler.c-9.us-east-1.aws.neon.tech`; error `Connection terminated unexpectedly` during extended batch |
| **Observation** | Remote pooled serverless endpoint; each Drizzle call is a network round trip; long-running cert sessions (~60+ min for 500 iterations) hit connection instability |
| **Impact** | p99 tail latency (17272 ms batch 100); batch 500/1000 interruption risk |
| **Recommendation** | Run large perf batches on isolated Neon branch; consider compute wake/cold-start; **no code change in 3C.2** |
