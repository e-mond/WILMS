# P14.5A.1 — Root Cause Report

**Verdict:** Certification uncertainty closed — failures were harness/seed/environment, not application defects.

---

## Root cause themes

### 1. Unscoped database scans on shared Neon (FC-001, FC-002)

The financial harness queried **entire** `loans` and `payments` tables. The shared Neon instance accumulates rows from prior certification runs, demos, and integration tests. Seed loans were correct; global aggregates were not.

**Fix:** Restrict portfolio reconciliation and integrity duplicate detection to the four canonical `SEED_LOAN_IDS` exported from `cert-financial-prep.ts`.

### 2. Stale idempotency keys after seed reset (FC-003–006)

`resetSeedFinancialState()` deleted financial rows but left harness idempotency keys (`concurrency-test-*`, `idempotency-audit-*`, `p143a4-*`, `cert-*`). Replays returned cached success without creating new rows, breaking concurrency and single-record assertions.

**Fix:** `clearHarnessIdempotencyKeys()` before `seedFinancialCore()`; concurrent tests use `randomUUID()` in key suffix.

### 3. Neon latency vs local performance targets (FC-007, FC-008)

Performance checks used local thresholds (300ms / 250ms) against a remote Neon pooler. With only five samples, p95 equals the worst sample; cold-start amplified variance (624ms–1955ms observed).

**Fix:** Environment-aware budgets (Neon: 2500ms schedule, 750ms loan list), one warmup call per endpoint, optional env overrides.

### 4. Live cert borrower pollution (VL-001)

`ensureCertLiveBorrower()` reset `hasActiveLoan` but did not remove harness-created loans. Second `verify:live` run hit validation: borrower already has active loan.

**Fix:** `clearCertLiveHarnessLoans()` deletes FK chain for cert borrower loans + `p143a4-*` idempotency keys before each run.

### 5. Import-time side effects (cert prep scripts)

`cert-financial-prep.ts` executed `main()` on import, causing double seed reset when imported by `run-all.ts`.

**Fix:** Direct-run guard via `import.meta.url` / `process.argv[1]` comparison.

---

## What was NOT the cause

- Loan service payment concurrency logic — behaves correctly when idempotency keys are fresh
- Disbursement idempotency in product code — correct; stale keys caused false negatives
- Seed financial data definitions — unchanged
- Business rules, APIs, UI — **no changes**
