## Why split deployment (not monolith)

| Factor | Evidence | Implication |
|--------|----------|-------------|
| Backend entry | `apps/backend/src/index.ts` — `app.listen()` | Requires **persistent** Node process |
| No serverless handler | No Lambda/Vercel adapter export | Cannot run Express on Vercel Functions as-is |
| Stateful middleware | In-memory rate limit, local upload fallback | Multi-instance needs external stores (future) |
| Frontend BFF | `apps/frontend/src/app/api/wilms/[...path]/route.ts` | Next.js **must** run on edge/server platform |
| Root `npm run build` | `package.json` L12 — frontend only | Backend has no compile step; starts via `tsx` |

**Monolithic single-container** (Option D) is technically possible with Docker + process manager but adds operational complexity without benefit for this stack.

## Component responsibilities

| Component | Host | URL pattern |
|-----------|------|-------------|
| Public UI | Vercel | `https://app.wilms.example` |
| BFF API | Vercel (Next routes) | `https://app.wilms.example/api/wilms/*` |
| Business API | Railway | `https://api.wilms.example` (or Railway default domain) |
| Database | Neon | `DATABASE_URL` (private connection string) |
| File storage | Cloudinary | `CLOUDINARY_*` env vars |

---

## Request flow (production)

1. Browser loads page from Vercel.
2. Client calls `NEXT_PUBLIC_API_BASE_URL` → same-origin `/api/wilms/...`.
3. Next.js BFF reads `wilms_session` cookie → adds `Authorization: Bearer`.
4. BFF proxies to `WILMS_API_UPSTREAM` (Railway Express).
5. Express validates HMAC token, enforces RBAC, queries Neon / Cloudinary.

**CORS:** Browser never calls Railway directly in BFF pattern; `WILMS_CORS_ORIGIN` still required for production validation and direct API tooling.
