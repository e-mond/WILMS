## 2. Proposed workflow architecture

```
feature/* → PR → CI (ci.yml) → merge main
                    ↓
            deploy-staging.yml (auto)
                    ↓
            smoke:staging (15+ checks)
                    ↓
            manual approval (GitHub Environment: production)
                    ↓
            deploy-production.yml
                    ↓
            Deployment Verification Pipeline
```

### `.github/workflows/ci.yml` (enhancements)

Add jobs/steps:

- `npm audit --audit-level=high` (fail on high/critical)
- `gitleaks` or `trufflehog` secret scan
- `npm run perf:budget-check` (new script — Lighthouse CI or custom)
- `npm run bundle:budget-check` (new script — parse build output)

### `.github/workflows/deploy-staging.yml` (new)

- Trigger: `push` to `main`, `workflow_run` after CI success
- Deploy Railway **staging** service + Vercel preview (or dedicated staging project)
- Run `npm run smoke:staging` with `STAGING_*` secrets
- Post job summary (pass/fail)

### `.github/workflows/deploy-production.yml` (replace)

- Trigger: `workflow_dispatch` only, **or** `workflow_run` after staging with `environment: production`
- GitHub Environment `production` with **required reviewers**
- Verification pipeline steps (sequential, hard-fail):

  1. Confirm old version healthy (`curl /health`)
  2. Deploy API (Railway) + frontend (Vercel prod)
  3. Health 200 + version match `package.json`
  4. `npm run smoke:production`
  5. Log/metrics spot-check (manual artifact or Railway/Vercel dashboard link)
  6. Mark complete

- **No automatic rollback** on failure

## 6. Hard-fail conditions (deploy pipeline)

Per P14.6 spec — any failure halts pipeline:

- lint, build, test, audit, secret scan failures
- smoke test failure (staging or production)
- migration failure
- `/health` ≠ 200
- version mismatch across UI + `/health` + `package.json`
- mock flag enabled in target env
- localhost in production network calls (smoke check)
- CSRF validation failure on smoke login
- perf/bundle budget exceeded (unless dated risk acceptance on file)
