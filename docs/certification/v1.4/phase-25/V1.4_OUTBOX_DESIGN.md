# V1.4 Outbox Design

**Date:** 18 July 2026  
**Pattern:** Transactional outbox (at-least-once delivery)  
**GL dual-write:** **NOT implemented** (v1.5 scope)

---

## Purpose

Provide durable domain event infrastructure so money mutations can record side effects in the same database transaction and deliver them asynchronously. Phase 25 delivers schema, writer, publisher, and queue delivery — **not** downstream GL consumers.

## Writer

**Path:** `apps/backend/src/infrastructure/outbox/writer.ts`

```typescript
writeOutboxEvent(input, tx?) → string | null
```

- No-op when `outboxEnabled` false or DB disabled
- Inserts `PENDING` row in caller's transaction
- Returns outbox row ID

**Current call sites:** Writer exported; **no money mutation paths invoke it yet** in this certification scope. Infrastructure is ready for incremental adoption.

---

## Publisher & delivery

**Path:** `apps/backend/src/infrastructure/outbox/publisher.ts`

### `publishPendingOutbox(limit=50)`

1. Select `PENDING` rows where `available_at <= now()`
2. For each row: `enqueueJob('outbox', { type: 'outbox.deliver', payload: {...} })`
3. Mark row `PROCESSING`; increment `attempts`
4. On enqueue failure: set `last_error`, backoff `available_at` + 30 s

### Job handler: `outbox.deliver`

1. Log `outbox.delivered` with `outboxId`, `eventType`, `correlationId`
2. Mark row `COMPLETED`, set `processed_at`

### Poller job: `outbox.poll`

Triggers `publishPendingOutbox()` — for scheduler/HTTP trigger integration.

**Delivery semantics:** At-least-once. Future consumers (e.g. GL dual-write) **must** be idempotent by `outboxId`.

## Feature flags

| Flag | Env | Default |
|------|-----|---------|
| `outboxEnabled` | `WILMS_FLAG_OUTBOX` | `true` |
| `outboxDelivery` | `WILMS_FLAG_OUTBOX_DELIVERY` | `true` when `REDIS_URL` set |
| `glDualWrite` | `WILMS_FLAG_GL_DUAL_WRITE` | **`false` always** |

When `outboxDelivery` false, rows accumulate as `PENDING` — no queue dispatch.

---

## Queue integration

| Queue name | Job type | Handler |
|------------|----------|---------|
| `outbox` | `outbox.deliver` | Mark completed, log delivery |
| `outbox` | `outbox.poll` | Batch publish pending rows |

Registered via `registerOutboxHandlers()` in `job-handlers.ts`.

## Data flow

```
┌─────────────────┐     same TX      ┌──────────────────┐
│ Money mutation  │ ───────────────► │ domain_outbox    │
│ (future)        │                  │ status=PENDING   │
└─────────────────┘                  └────────┬─────────┘
                                            │
                              publishPendingOutbox()
                                            │
                                            ▼
                                   ┌──────────────────┐
                                   │ BullMQ / in-proc │
                                   │ outbox.deliver   │
                                   └────────┬─────────┘
                                            │
                                            ▼
                                   status=COMPLETED
                                   (no GL write today)
```
