# V1.4 Idempotency Report

**Date:** 18 July 2026  
**Scope:** Money mutation idempotency hardening on existing `idempotency_keys` table

## Summary

Phase 25 enhances idempotency with **request payload hash conflict detection** (HTTP 409 `IDEMPOTENCY_KEY_REUSED`) and makes `Idempotency-Key` **required by default in production** via the `WILMS_FLAG_REQUIRE_IDEMPOTENCY` feature flag. Scope enum values are **unchanged**.

## Phase 25 behavior

| Scenario | HTTP | Error code |
|----------|------|------------|
| Missing key when `requireIdempotency` true | 400 | `IDEMPOTENCY_REQUIRED` |
| Same key, same payload (completed) | 200 | Replay cached body |
| Same key, different payload hash | **409** | **`IDEMPOTENCY_KEY_REUSED`** |
| Same key, concurrent in-flight | 409 | `IDEMPOTENCY_IN_PROGRESS` |
| Key &gt; 256 chars | 400 | `VALIDATION` |

**Implementation:** `apps/backend/src/infrastructure/idempotency/run-with-idempotency.ts`  
**Repository:** `apps/backend/src/repositories/idempotency.repository.ts`

Hash function: `SHA-256(JSON.stringify(payload))` via `hashIdempotencyPayload()`.

## Client guidance

1. Generate a UUID v4 (or equivalent) per logical money operation.
2. Send as `Idempotency-Key` header on POST mutations.
3. On network timeout, **retry with the same key** and identical body.
4. On 409 `IDEMPOTENCY_KEY_REUSED`, treat as client bug (key reused with different payload).

---

## Conclusion

Idempotency hardening is **complete** at the API layer with tests and production-default enforcement via feature flag. End-to-end production certification requires deploy and client compliance verification.
