# V1.4 BullMQ Implementation Report

**Date:** 18 July 2026  
**Packages:** `bullmq` ^5.80.8, `ioredis` ^5.11.1  
**Mode:** Optional durable queues with **in-process fallback**

---

## Architecture

```
enqueueJob()
    │
    ├─ durableQueues flag ON + REDIS_URL + Redis reachable
    │       └─ BullMQ (mail, sms, export, scheduler, outbox queues)
    │
    └─ otherwise
            └─ in-process immediate execution (non-durable)
```

**Source:** `apps/backend/src/infrastructure/queue/index.ts`

## Configuration

| Variable | Purpose | Default |
|----------|---------|---------|
| `REDIS_URL` or `WILMS_REDIS_URL` | Redis connection | Unset → in-process |
| `WILMS_QUEUE_PREFIX` | BullMQ queue name prefix | `wilms` |
| `WILMS_FLAG_DURABLE_QUEUES` | Enable BullMQ when Redis available | `true` when `REDIS_URL` set |

Queue names: `mail`, `sms`, `export`, `scheduler`, `outbox` (plus `dlq` type reserved).

---

## BullMQ job defaults

| Option | Value |
|--------|-------|
| Attempts | 5 |
| Backoff | Exponential, 2000 ms base |
| `removeOnComplete` | Keep 1000 |
| `removeOnFail` | Keep 5000 |
| Worker concurrency | 5 per queue |

Connection timeout: 3 s; retry strategy caps at 3 attempts then fails over.

---

## Lifecycle

| Event | Behavior | Source |
|-------|----------|--------|
| Startup | `registerBuiltInJobHandlers()` then `startQueueWorkers()` | `src/index.ts` |
| Shutdown | `stopQueueWorkers()` before `server.close()` on SIGTERM/SIGINT | `src/index.ts` |
| Redis connect failure | `connectionFailed = true`; fallback to in-process | `bullmq-queue.ts` |
| Enqueue failure | Log `queue.fallback_in_process`; run in-process | `queue/index.ts` |

## Known limitations

1. **In-process default:** Without Redis, jobs are not durable across restarts.
2. **No DLQ consumer UI:** Failed jobs logged; admin replay not implemented.
3. **Shared process:** Workers share the API Node process — acceptable for v1.4 foundation, not final ops topology.

---

## Conclusion

BullMQ integration is **implemented and tested** for fallback path and ops reporting. **Durable queue operation in staging/production is not certified** in this phase — requires operator Redis provision and post-deploy verification.
