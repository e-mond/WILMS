## Deployment Guide

### Prerequisites

- Node.js 20+
- PostgreSQL (Neon recommended)
- Cloudinary account (production uploads)
- Hosting for Next.js frontend + Node.js API (or combined BFF)

### 1. Database

```bash
cd apps/backend
# Set DATABASE_URL in environment
npm run db:migrate -w @wilms/api
npm run db:seed -w @wilms/api
```

### 2. Backend API

```bash
# Set variables from apps/backend/.env.production.example
export NODE_ENV=production
export DATABASE_URL='postgresql://...'
export UPLOAD_PROVIDER=cloudinary
export CLOUDINARY_CLOUD_NAME='...'
export CLOUDINARY_API_KEY='...'
export CLOUDINARY_API_SECRET='...'
export WILMS_CORS_ORIGIN='https://app.example.com'

npm run start -w @wilms/api
```

Verify:

```bash
curl -s https://api.example.com/health
```

### 3. Frontend

```bash
# Set variables from apps/frontend/.env.production.example
export NODE_ENV=production
export NEXT_PUBLIC_APP_URL='https://app.example.com'
export NEXT_PUBLIC_API_BASE_URL='https://app.example.com/api/wilms'
export WILMS_API_UPSTREAM='https://api.example.com'

npm run build -w @wilms/frontend
npm run start -w @wilms/frontend
```

### 4. Post-deploy smoke

```bash
npm run cert:upload:env -w @wilms/api
npm run verify:live -w @wilms/api
node apps/frontend/scripts/proxy-reconciliation-smoke.mjs
```

## Rollback Guide

### Application rollback

1. Redeploy previous container/image tag for frontend and backend.
2. No automatic DB downgrade ÔÇö migrations are forward-only.

### Database rollback

**Do not** run destructive rollback on production without backup restore.

1. Restore Neon branch from point-in-time backup (Neon console).
2. Or: deploy previous app version compatible with current schema (preferred if migration already applied).

### Upload rollback

- Cloudinary assets persist independently; deleting app deployment does not remove CDN objects.
- Use Cloudinary console or `POST /uploads/:id/delete` for individual cleanup.

### Session impact

- Users must re-login after backend rollback if session signing changes.

## Database Migration Guide

### Migration chain (apply in order)

```
0000_init ÔåÆ 0001_tired_sir_ram ÔåÆ 0002_sad_bucky ÔåÆ 0003_loan_pools
ÔåÆ 0004_financial_adjustments ÔåÆ 0005_financial_reversals ÔåÆ 0006_financial_reconciliations
```

### Apply migrations

```bash
cd apps/backend
npm run db:migrate -w @wilms/api
```

### Verify

```bash
npm run cert:reversal:env -w @wilms/api
npm run cert:reconciliation:env -w @wilms/api
```

### Seed (staging only)

```bash
npm run db:seed -w @wilms/api
```

**Warning:** Do not re-seed production.

## Operational Checklist

### Daily (demo/staging)

- [ ] `GET /health` returns 200
- [ ] Demo user login works
- [ ] Collector reconciliation page loads (live API)

### Pre-demo

- [ ] `NEXT_PUBLIC_USE_MOCK=false` (dev) or production env set
- [ ] Proxy smoke 5/5
- [ ] Recovery guide available (`P14.3B-demo-recovery-guide.md`)

### Pre-release

- [ ] All migrations applied
- [ ] `npm run type-check && npm run lint && npm run build && npm run test`
- [ ] `cert:demo:stakeholder` ÔëÑ30/31 effective
- [ ] Cloudinary upload test on staging
- [ ] Risk register blocking items resolved

### Incident response

| Symptom | Action |
|---------|--------|
| 401 on API | Check session cookie / Bearer token |
| 403 on reconciliation | Verify `RECORD_COLLECTIONS` role |
| Upload 500 | Check Cloudinary creds; verify UUID fix deployed |
| Mock data in UI | Set `NEXT_PUBLIC_USE_MOCK=false`; restart frontend |
