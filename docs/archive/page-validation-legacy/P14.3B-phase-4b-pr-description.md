## Architecture Decisions

| ID | Decision |
|----|----------|
| AD-REC-01 | v1 = collector cash reconciliation only (not pool/branch rollup) |
| AD-REC-02 | Primary variance = `physical_cash ÔêÆ expected_due` |
| AD-REC-03 | System recorded = CONFIRMED payments only (exclude REVERSED) |
| AD-REC-04 | Defer paymentÔåÆpool REPAYMENT allocation to Pool Hardening |
| AD-REC-05 | Defer reversal pool compensating entries to Pool Hardening |
| AD-REC-06 | Tables: `financial_reconciliations` + `reconciliation_history` |
| AD-REC-07 | Unique `(collector_user_id, reconciliation_date)` |
| AD-REC-08 | Reuse `RECORD_COLLECTIONS`; no new RBAC permissions v1 |
| AD-REC-09 | Submit-only workflow in 4C (no approval state machine) |
| AD-REC-10 | Wire `RECONCILIATION_SUBMITTED` audit enum in 4C |

---

## Findings

1. Frontend reconciliation **complete on mock** ÔÇö GET `/reconciliation`, POST `/reconciliations`
2. Backend has **zero** reconciliation implementation
3. Mock variance uses `physical ÔêÆ actual`; architecture aligns primary variance to `physical ÔêÆ expected`
4. `REVERSED` payments must be excluded from system recorded sum
5. Pool allocations and reversal pool entries **deferred** ÔÇö orthogonal to collector cash reconcile
6. Audit enums exist but `audit.repository.ts` ACTION_MAP not wired
