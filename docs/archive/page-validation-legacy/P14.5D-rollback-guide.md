## Database rollback (Neon)

| Scenario | Action |
|----------|--------|
| Failed migration on deploy | Fix migration SQL; run `npm run db:migrate -w @wilms/api` from fixed code |
| Data corruption | Restore Neon **branch** or **point-in-time recovery** from Neon console |
| Accidental seed in prod | **Never** run `db:seed` in production without approval |

**Pre-deploy:** Create Neon backup branch before first production migration.

## Cloudinary recovery

| Issue | Action |
|-------|--------|
| Upload 500 | Verify `CLOUDINARY_*` on Railway; check `/health` uploads section |
| Wrong folder | Update `CLOUDINARY_FOLDER`; new uploads use new path |
| Credential rotation | Update Railway env; redeploy; no code change |

## Incident runbook ÔÇö production down

1. **Detect:** Uptime monitor or user report
2. **Triage:** `curl /health` on API; check Vercel deployment status
3. **Frontend only broken:** Vercel rollback (above)
4. **API only broken:** Railway rollback; check Neon connectivity
5. **Database:** Neon dashboard ÔåÆ connection limits, branch status
6. **Communicate:** Status page / stakeholders
7. **Post-incident:** Document in `docs/page-validation/` with timestamps and deployment IDs
