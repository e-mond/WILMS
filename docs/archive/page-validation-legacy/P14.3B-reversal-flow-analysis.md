## Summary

Four primary Neon-backed financial write flows were traced. A fifth path (`PATCH /payments`) exists but **does not perform financial reversal** ÔÇö recorded as risk **R-FLOW-05**.

| Flow | Atomic txn | Ledger | Balance | Audit | Idempotency | Pool allocation |
|------|------------|--------|---------|-------|-------------|-----------------|
| Payment posting | Yes | REPAYMENT | Yes | After commit | PAYMENT_POST | **No** |
| Adjustment approve | Yes | ADJUSTMENT | Yes | After commit | ADJUSTMENT_APPROVE | Yes (if pool linked) |
| Pool allocation (runtime) | Part of adjustment txn | N/A | Denormalized pool cols (seed only) | N/A | N/A | Append-only insert |
| Loan disbursement | Yes | LOAN_DISBURSEMENT | Via lifecycle | After commit | LOAN_DISBURSE | **No** |

## 1. Payment Posting

### Entry

| Item | Value |
|------|-------|
| **File** | `apps/backend/src/modules/payments/routes.ts` |
| **Route** | `POST /payments` |
| **Permission** | `RECORD_COLLECTIONS` |
| **Service** | `paymentService.recordPayment` ÔåÆ `postPayment` |

### Function chain

```text
recordPayment (L113)
  ÔööÔöÇ runWithIdempotency { scope: PAYMENT_POST } (L120-126)
       ÔööÔöÇ postPayment (L129-253)
            ÔööÔöÇ runInTransaction (L179-232)
```

### Transaction boundary

Single `runInTransaction` block writes:

1. `payments` row ÔÇö `paymentRepo.appendPayment`
2. Schedule week ÔåÆ `PAID` ÔÇö `scheduleRepo.markWeekPaid` (optimistic `weekRow.version`)
3. Loan balance + lifecycle ÔÇö `loanRepo.updateLoanLifecycle` (optimistic `activeLoan.version`)
4. Ledger ÔÇö `ledgerRepo.appendLedgerEntry` type `REPAYMENT` with `paymentId`

**Evidence:**

```179:229:apps/backend/src/modules/payments/service.ts
  const result = await runInTransaction(async (tx) => {
    const payment = await paymentRepo.appendPayment({ ... }, tx);
    await scheduleRepo.markWeekPaid({ loanId, weekNumber, expectedVersion: weekRow.version }, tx);
    await loanRepo.updateLoanLifecycle({ loanId, expectedVersion: activeLoan.version, lifecycleStatus, loanBalance }, tx);
    await ledgerRepo.appendLedgerEntry({ entryType: 'REPAYMENT', paymentId: payment.id, ... }, tx);
    return payment;
  });
```

### Ledger write

| Field | Value |
|-------|-------|
| Type | `REPAYMENT` |
| Repository | `ledger.repository.appendLedgerEntry` |
| Link | `paymentId` FK populated |

### Audit write

| Item | Value |
|------|-------|
| When | **After** transaction commit |
| Action | `payment.recorded` ÔåÆ `PAYMENT_RECORDED` |
| Pattern | Best-effort async (`audit-log.ts`) |

```234:239:apps/backend/src/modules/payments/service.ts
  appendAuditEntry({
    action: 'payment.recorded',
    actorId,
    targetEntityId: result.id,
    targetEntityType: 'payment',
  });
```

### Balance update

`loanBalance = max(outstanding - amount, 0)`; lifecycle ÔåÆ `COMPLETED` if zero else `ACTIVE`.

### Idempotency

| Scope | `PAYMENT_POST` |
| Header | `Idempotency-Key` on route (L135) |
| Wrapper | `run-with-idempotency.ts` ÔÇö replay on `COMPLETED` |

### Pool allocation

**Not written** on payment post. Grep shows `appendAllocation` only in seed + adjustment approve ÔÇö not in `payments/service.ts`.

### Reversal design implication

Payment reversal must undo **four coupled writes** inside one transaction: payment status, schedule week, loan balance/lifecycle, compensating ledger entry, and optionally pool `REPAYMENT` allocation when disburse hooks exist.

### Entry

| Item | Value |
|------|-------|
| **File** | `apps/backend/src/modules/adjustments/routes.ts` |
| **Route** | `POST /adjustments/:id/approve` |
| **Permission** | `ACCESS_ADMIN_PORTAL` |
| **Service** | `approveAdjustment` |

### Function chain

```text
approveAdjustment (L190)
  ÔööÔöÇ runWithIdempotency { scope: ADJUSTMENT_APPROVE }
       ÔööÔöÇ runInTransaction (L225-334)
```

### Transaction boundary (approve)

1. `financial_adjustments` status ÔåÆ APPROVED + before/after/delta ÔÇö `approveAdjustmentRow` (version lock)
2. `loans` balance + lifecycle ÔÇö `updateLoanLifecycle`
3. `ledger_entries` type `ADJUSTMENT` ÔÇö metadata includes `adjustmentId`, deltas
4. `pool_allocations` type `ADJUSTMENT` ÔÇö if `loan.loanPoolId` and delta Ôëá 0
5. Borrower blacklist ÔÇö if `WRITE_OFF`
6. `adjustment_history` ÔÇö APPROVED + LEDGER_POSTED events

**Evidence (pool hook):**

```280:292:apps/backend/src/modules/adjustments/service.ts
          if (loan.loanPoolId && effect.deltaPesewas !== 0) {
            await poolRepo.appendAllocation({
              poolId: loan.loanPoolId,
              allocationType: 'ADJUSTMENT',
              amountPesewas: Math.abs(effect.deltaPesewas),
              ...
            }, tx);
          }
```

### Ledger write

Type `ADJUSTMENT` ÔÇö not linked to a prior ledger row ID (no `reversesEntryId` column today).

### Audit write

After commit: `adjustment.approved` ÔåÆ `ADJUSTMENT_APPROVED`.

### Idempotency

Scope `ADJUSTMENT_APPROVE`; header `idempotency-key` on route.

### Reversal design implication

Reversing an **approved adjustment** requires compensating ledger + pool rows linked to original adjustment ledger entry; must block if source payment already reversed.

## 3. Pool Allocation (Runtime)

### Writers today

| Source | File | Types |
|--------|------|-------|
| Seed | `db/seed/seed-loan-pools.ts` | DISBURSEMENT, REPAYMENT, REPLENISHMENT |
| Adjustment approve | `adjustments/service.ts` | ADJUSTMENT |

### Function

`loan-pool.repository.appendAllocation` ÔÇö **insert only**, no update/delete.

```110:137:apps/backend/src/repositories/loan-pool.repository.ts
export async function appendAllocation(...) {
  await tx.insert(poolAllocations).values({ ... });
}
```

### Transaction boundary

- Seed: per-pool `runInTransaction` in seed script
- Runtime: sub-step of adjustment approve transaction

### Ledger / audit / idempotency

Pool allocations have **no** direct ledger row or audit entry today.

### Reversal design implication

Pool reversal = **compensating allocation row** with link metadata to source allocation ID; optional enum value `REVERSAL` or reuse `ADJUSTMENT` with `reversesAllocationId`.

### Pool
