## Domain Model (approved for Phase 1)

```text
loan_pools          Capital bucket (region, source, aggregates)
    Ôöé
    Ôö£ÔöÇÔöÇ pool_memberships ÔöÇÔöÇÔû║ groups (many groups per pool)
    Ôöé
    ÔööÔöÇÔöÇ pool_allocations ÔöÇÔöÇÔû║ loans / payments (append-only ledger of pool movements)

loans.loan_pool_id  Nullable FK ÔÇö set when disbursement is attributed to a pool
```

**Ownership:** Pools are **organizational funding instruments**, not collector-owned. Collectors remain user extensions (ADR-001); they do not own pool capital.

## Classification Matrix

| Concern | Classification | Evidence / decision |
|---------|----------------|---------------------|
| **Pool ownership** | **Safe** | Super Admin / financial reporting roles read pools. No collector write access in frontend. Backend: `VIEW_FINANCIAL_REPORTS` on GET routes (matches `/loans` read pattern). |
| **Collector ownership** | **Safe** | ADR-001: `payments.collector_user_id ÔåÆ users.id`. Repayments attributed to collector on payment row; pool attribution is separate `pool_allocations` row with `REPAYMENT` type. |
| **Borrower membership** | **Needs Alignment** | Borrowers join pools **indirectly** via `pool_memberships.group_id ÔåÆ group_members`. No direct borrowerÔåöpool table (avoids duplicate membership truth). Phase 1 seeds group links only. |
| **Allocation rules** | **Safe (Phase 1 read-only)** | `pool_allocations` append-only: `DISBURSEMENT`, `REPAYMENT`, `REPLENISHMENT`, `ADJUSTMENT`. Aggregates on `loan_pools` derived from allocation sums. Disburse hook deferred ÔÇö no change to P14.3A disburse transaction yet. |
| **Repayment attribution** | **Needs Alignment** | Payment service does not yet write `REPAYMENT` pool allocations. Phase 1 seeds historical allocations; Phase 4 reconciliation will require paymentÔåÆpool linkage. |
| **Reconciliation impact** | **Needs Alignment** | Reconciliation compares collector cash vs **payments**, not pool balances. Pool totals must reconcile to sum(allocations) ÔÇö separate harness check in `verify:pools`. |
| **Write-off impact** | **Breaking Risk (future)** | Write-off reduces borrower obligation but pool `outstanding_pesewas` must decrease via `ADJUSTMENT` allocation, not loan row mutation alone. Deferred to Phase 5. |
| **Adjustment impact** | **Breaking Risk (future)** | Balance corrections must post compensating `pool_allocations` of type `ADJUSTMENT`. Deferred to Phase 2. |
| **Loan list FK** | **Needs Alignment** | `loans.loan_pool_id` nullable added in migration; optional until disburse integration. No breaking change to existing loans. |
| **Frontend contract** | **Safe** | `GET /loan-pools`, `GET /loan-pools/:id` match `ILoanPoolService`. Response shape matches `LoanPoolListResponse` / `LoanPoolDetail` (`{ data }` envelope). |
| **Mock ID vs UUID** | **Needs Alignment** | Mock uses `PL-001` strings; backend uses UUID v7 (consistent with P14.3A). UI accepts any string id ÔÇö no breaking change. |
| **totalPoolFunds KPI** | **Needs Alignment** | Mock uses fixed `DEMO_OPERATING_POOL_PESEWAS` (482M) Ôëá sum of pool capitals. API returns **sum of pool capital** for production honesty; documented in Phase 1 report. |

## Allocation Rules (Phase 1)

1. **DISBURSEMENT** ÔÇö Increases `disbursed_pesewas`; requires `loan_id`, `amount_pesewas > 0`.
2. **REPAYMENT** ÔÇö Increases `collected_pesewas`; may reference `loan_id` and payment metadata.
3. **REPLENISHMENT** ÔÇö Increases `capital_pesewas` (funding top-up).
4. **ADJUSTMENT** ÔÇö Reserved for Phase 2; enum present, no writer in Phase 1.

**Balance invariants (verified by `verify:pools`):**

```text
disbursed_pesewas  = ╬ú DISBURSEMENT amounts
collected_pesewas  = ╬ú REPAYMENT amounts
outstanding_pesewas = disbursed_pesewas ÔêÆ collected_pesewas
utilisation_percent = round(disbursed / capital ├ù 100)  [capped 0ÔÇô100]
repayment_rate_percent = round(collected / disbursed ├ù 100)  [when disbursed > 0]
```

All aggregate updates run inside transactions when allocation writers land (Phase 1: seed-only writes).
